# Finspot — Claude Code Rules

## Project

A dive/snorkel logging app built step-by-step as a portfolio and learning project.
Goal: learn full-stack concepts properly, not just ship features.

## Guiding Principle

When choosing whether to add something, ask: "Would a professional team use this, and would understanding it help in an interview?" If yes, add it — even if it's not strictly necessary for a project this size. This is a learning project first, portfolio second.

## Who I Am & Why This Matters

- Frontend developer with strong frontend skills, currently doing full-stack work
- Building FinSpot to deliberately strengthen backend and full-stack knowledge
- Target: land a full-stack role at ~$130k AUD — need to hold my own in a 90-minute concept-heavy interview
- I learn best by doing. I want to understand and explain every decision, not just have AI generate code
- Key interview topics I need to be strong on: schema design, relationships, migrations, auth, CRUD, API design, system design trade-offs, server vs client components
- This is primarily a learning vehicle, secondarily a portfolio piece

## Teaching Focus

When explaining concepts, frame them in terms of:

- "How would you explain this in an interview?"
- "Why did we choose this over the alternatives?"
- "What would a senior engineer say about this decision?"
- Connect backend concepts to frontend analogues where possible (I already know the frontend well)

## Hard Rules

### 🚫 Never commit without explicit permission

- NEVER run `git add`, `git commit`, or `git push` unless the user explicitly says
  something like "commit this", "go ahead and commit", or "push it"
- After making changes, list the files that changed and STOP
- Let the user review in Cursor before anything is committed

### 🐢 Explain before building

- Before writing any code, explain: what we're building, why, and how it works
- Wait for the user to confirm ("ok", "go ahead", "do it") before implementing
- If a concept is new (e.g. server components, ORMs, middleware), explain it first

### 🔂 One step at a time

- Make one atomic change per turn (e.g. install a package OR create one file — not both)
- After each change, list the files affected and STOP
- Never chain multiple features together without checking in between
- Treat each change as its own commit — small, reviewable, purposeful

### 🎓 Teaching mode

- After completing each feature, ask 3–5 interview questions ONE AT A TIME
- Do NOT provide the answer when asking the question — wait for the user to answer first
- After the user answers, give feedback, correct any gaps, and provide a model answer
- Then ask "Ready for the next one?" before moving on
- Call out industry-standard patterns when they appear
- Highlight "why we do it this way" not just "what to type"

### 📚 Always use current documentation

- Before installing or configuring any package, check the official docs for the latest stable version
- If unsure whether guidance is current, say so and ask the user to find the relevant docs page
- Never rely on training knowledge alone for fast-moving tools (Prisma, Next.js, Better Auth, shadcn, etc.)
- When the user links to a docs page, treat it as the source of truth over anything else
- Context7 MCP is configured globally — it will automatically fetch live docs for any library

### 🏭 Production quality by default

- Always implement things the way a professional production app would, not the simplest possible shortcut
- If a simpler approach is being used for a legitimate reason (e.g. scope, time), say so explicitly
- If something would be done differently in a real company codebase, flag it — e.g. "this works but in production you'd typically..."
- Examples of things to flag: storing arrays in a single column instead of a relation, skipping input validation, hardcoding values that should be config, using patterns that don't scale

## Confirmed Stack

- Framework: Next.js (App Router) + TypeScript
- Styling: Tailwind CSS v4 + shadcn/ui
- Database: PostgreSQL on Neon
- ORM: Prisma
- Auth: Better Auth
- File storage: Cloudinary or S3 - undecided
- Deploy: Vercel

## Project Structure (will grow over time)

- `app/` — Next.js pages and layouts (App Router)
- `components/ui/` — shadcn/ui components (generated by shadcn, can be edited)
- `lib/` — shared utilities (e.g. db client, auth config)
- `prisma/` — schema and migrations

## Current Phase: Phase 1 — Authentication (Better Auth)

### Concepts to cover

1. How web auth works — cookies, sessions, what "being logged in" means
2. What a JWT is — the idea behind signed tokens, even though Better Auth handles it
3. What middleware is — how Next.js intercepts requests before they hit a page
4. Protected routes — the pattern for blocking unauthenticated users

### Build steps

- [ ] Install Better Auth and dependencies
- [ ] Create `lib/auth.ts` — core config (providers, database connection)
- [ ] Run Better Auth's migration — adds `user`, `session`, `account` tables
- [ ] Create `app/api/auth/[...all]/route.ts` — API route handler
- [ ] Create `lib/auth-client.ts` — client-side auth helper
- [ ] Build sign up page
- [ ] Build sign in page
- [ ] Set up middleware for protected routes
- [ ] Test with a protected page
